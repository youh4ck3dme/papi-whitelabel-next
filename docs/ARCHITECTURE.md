# Architecture Overview

## System Overview
A Next.js application providing booking and billing APIs for a multi-tenant SaaS use case. The current implementation is backend-hardening focused, with frontend flows partially scaffolded.

## Boundaries
- Frontend: Next.js routes/UI for tenant/admin flows.
- API: Next.js route handlers under `src/app/api/*`.
- Data: Prisma + database via `DATABASE_URL`.
- Payments: Stripe server-side APIs + webhook ingestion.
- Optional integrations: Twilio, Zapier, OpenAI, NOWPayments (non-core; should degrade safely).

## Tenant Isolation Model
- Authenticated user context provides tenant scope.
- Mutating routes enforce role and tenant checks.
- Client-provided tenant identifiers are not blindly trusted.

## Request Lifecycle (Mutating API)
1. Correlation/request ID attached in middleware.
2. Auth + role guard checks.
3. Tenant context validation.
4. Schema validation with stable error envelope.
5. Domain logic execution.
6. Structured audit/security logging.

## Payment Lifecycle
1. Client initiates payment/subscription request.
2. Server validates payload and tenant authorization.
3. Stripe interaction occurs server-side with secret key.
4. Webhook events are signature-verified and idempotently processed.
5. Payment state transitions are checked against allowed transitions.

## Reliability Controls
- Retry with backoff for transient notification/integration failures.
- Dead-letter capture for repeated failures.
- Request timeout guard for long-running handlers.
- Health endpoint reports app/database readiness shape.

## Failure Modes
| Failure Mode | Expected Behavior | Mitigation |
| --- | --- | --- |
| Missing required env | Stable config/runtime error | Config validation + docs |
| Stripe webhook replay | Duplicate ignored/safely handled | Webhook idempotency guard |
| External integration outage | Core booking flow remains available | Optional integration fallback |
| Cross-tenant request | Request rejected | Tenant guard enforcement |
| High request burst | Stable `429` with retry hint | Rate limiting middleware |

## Known Tradeoffs
- DB-native overlap enforcement is provided as a Postgres SQL rollout script at `docs/sql/booking-no-overlap-active.sql`; Prisma schema DSL does not model this exclusion constraint directly, so rollout is operational rather than generated by Prisma migrations.
- Some product UX flows remain scaffold-level while API security controls are prioritized.
- Optional integrations are present in codebase but not required for core local setup.
